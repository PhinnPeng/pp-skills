## 前言：什么时候需要写PRD

不是所有产品设计都需要用 PRD 文档来呈现。根据需求复杂度，可分为三种情况：

| 类型 | 场景 | 说明 |
|------|------|------|
| **Index Design** | 简单改动（文字、图片、颜色、logo等） | 在任务管理系统（如 Jira）里说清楚需求即可，无需创建独立文档 |
| **Story Design** | 面向一个用户故事/用户场景的设计 | 日常最常见的PRD形式，也是本文重点 |
| **Watching Design** | 从无到有做一个大型产品（如电商、抖音、小红书） | 需要多产品经理协作，需要顶层PRD作为总纲，不会讲细节产品设计 |

> **核心观点**：随着开发节奏越来越快，简单需求不需要创建PRD文档，直接在成熟的项目管理/迭代管理工具里写清楚即可。

## PRD的六大组成部分

一份合格的PRD采用**金字塔结构**，内容层层递进，循序渐进地展开。

```
PRD结构（金字塔）
├── 版本说明
├── 背景与目标
├── 故事介绍
│   ├── 场景描述
│   ├── 价值分析
│   ├── 用户核心路径
│   ├── 漏斗目标
│   └── 路径规划（选填）
├── 概要设计
│   ├── 模块图
│   ├── 功能清单图
│   └── 页面结构图
├── 详细设计
└── 交付设计
    ├── 数据分析/买点设计
    └── 上线筹备
```

### 一、版本说明

记录每次更新的内容，但需要注意：

- **不是每次迭代写一次**，而是跟着 Review 会议走
- 每次 Review 后发生的变更需要记录
- 下次参加 Review 的同学能了解：相较上次，哪些改变了、哪些没改变、发生了哪些改变

---

### 二、背景与目标

**背景**：解释为什么公司需要这个功能，在业务上能带来什么

**目标**：为产品设定具体可衡量的指标

> **示例（社区场景）**：
> - **背景**：社区之前只有图文上传功能，导致视频博主被拒之门外，需要增加视频上传功能
> - **目标**：
>   - 每天至少100个用户发布视频内容
>   - 视频内容占整体浏览量的25%

> **提示**：目标部分不需要展开介绍如何达成，这只是一个预期/印象。

### 三、故事介绍

这是**很多产品新人会遗漏的部分**。很多新人直接打开 Axure 等工具开始画原型，忽略了解释产品意义。

#### 3.1 场景描述（Story）

将用户会做的事情用一句话一行列出来，通常一个PRD十几行即可。

**为什么这样列？**
- 让读者大概知道要做什么功能
- 虽然还不理解细节和限制条件，但脑海里能反映出竞品和自己使用过的产品
- 形成**画面感**非常重要

#### 3.2 价值分析

解释这个功能为什么能达成目标。

> **示例**：视频内容增多后，读者浏览结构会变化，用户在视频页停留时间更长（视频最长可达15分钟），从而提升整体 App 访问时长。

> **注意**：
> - 实际业务分析很复杂，需要产品经理、产品运营等多岗位协同
> - 不要孤军奋战，避免PRD逻辑漏洞

#### 3.3 用户核心路径

对应**产品设计五张图**的第一张图：用例图/用户核心路径图

> **示例（发布者路径）**：选择视频 → 填写视频描述 → 提交审核 → 审核通过 → 发布成功

#### 3.4 漏斗目标

每个箭头/步骤对应一个转化率指标：

> **示例**：
> - 用户点击发布意向到真正提交视频的提交比例
> - 提交后实际过审的过审比例

**为什么要做漏斗目标？**

| 特点 | 说明 |
|------|------|
| 与产品设计强相关 | 业务目标和运营手段存在不确定性，但产品漏斗目标与你的设计直接相关 |
| 代表预测能力 | 这是产品经理预测能力的体现 |
| 练习与积累 | 被结果打脸或证实都会让你更有信心，逐步积累经验 |
| 指导运营 | 告诉运营需要准备多少素材（如：要保证每天100条内容，可能需要准备200条素材） |

> 干了十几年的产品经理，看到产品设计，78九不离十能猜出每一步的点击率、转换率。

#### 3.5 路径规划（选填）

中长期规划，向团队传递信号，让技术架构具备兼容性：

- 不一定是确定要做的功能
- 不需要真正设计出来
- **目的是提前告知**，让技术经理在设计数据结构或代码架构时考虑未来兼容性
- **大大提高后续版本的开发效率**

> 这是体现产品经理实力和经验的环节。年轻产品经理可能无法规划太远，但要尝试，即使只写在自己本子上，也是锻炼过程。

### 四、概要设计

对应**产品设计五张图**中的三张核心图。

#### 4.1 模块图（可选）

展示产品模块的划分关系。

#### 4.2 功能清单图

用穷举法列出每个模块下会发生的功能，类似详细设计的目录。

**命名规则示例**：
- `M` 开头代表模块（Module）
- `F` 开头代表功能（Feature）

**为什么重要？**
1. 给团队全貌，让大家知道在干什么
2. Tech Lead 根据功能清单分配任务（如：两个工程师负责这个模块的三个功能）
3. 类似索引，保证任务不被遗漏
4. 测试团队根据功能清单准备 Test Case

#### 4.3 页面结构图

更直观的索引，与功能清单相互呼应。

**自洽检验**：
- 如果某个页面在功能清单里没有任何功能 → 页面多余？功能清单漏了？
- 如果功能清单里有功能但页面结构里没有承载 → 漏页面了？

> **建议**：至少用两种索引方式帮助产品设计自洽。在实际工作中，可以在这个阶段与交互设计师协作，使用墨刀、蓝湖等工具画出低保真原型。

---

### 五、详细设计

相对最简单最不容易出错的部分：将交互图中每个页面的逻辑一一对应讲清楚。

**关键点**：
- 详细设计要与功能清单、页面结构一一呼应
- 再次自洽检验

### 六、交付设计

这是**很多产品新人经常遗漏的部分**。

#### 6.1 数据分析 / 买点设计

**常见问题**：产品上线后发现忘记买点，数据取不到，只能等一两周再上线一次。

**解决方案**：在PRD模板中强制自己思考：
1. 这个产品需要数据分析吗？
2. 现有的买点设计能否满足未来的分析诉求？
3. 如果不能满足，需要在PRD里写清买点设计内容

**买点设计要素**（通常是一张表格）：

| 要素 | 说明 |
|------|------|
| Action | 触发的用户行为 |
| View | 页面/模块 |
| Code | 埋点编码 |

**数据需求设计**：
- 在产品未进研发之前就想好要看什么数据
- 不要等产品上线后再捞数据
- 可以画一张 Excel 报表附在PRD里，或描述清楚筛选条件和结果字段
- 记得在价值分析章节里提到的数据，都要在数据分析设计里想明白

**面向对象**：
- **工程师**：负责埋点实现
- **数据分析师**：了解要拿到什么数据；在测试阶段也是"用户"，验证买点是否正确

> **双保险机制**：如果数据分析师在测试阶段发现买点无法生成预期报表，产品测试不通过。

#### 6.2 上线筹备

偏技术型的工作，需要考虑：

| 事项 | 说明 |
|------|------|
| 历史数据处理 | 功能是否影响历史数据，是否需要备份/处理 |
| 依赖关系 | 是否依赖公司中台其他功能上线？如果对方没上线，我方不能提前上线 |

### 七、产品上线后的迭代更新

**毛毛的个人习惯**：上线后一两周数据稳定后，用分析能力更新整个PRD。

**更新内容**：
- 当初想要的目标价值，真实情况如何？
- 当初预测的漏斗数据，实际转化率如何？

> **这是一个非常牛逼的事情**：如果你设计10个PRD，10个PRD对应功能上线两周后的数据都跟你的预测很接近，那你绝对是专家。

**建议**：
- 不要怕刚开始预测不准、统计不准
- 通过循序渐进让自己变准
- 走出第一步，你就已经超过了很多人

## PRD Review 会议效率技巧

**核心方法**：提前发 PRD！

| 错误做法 | 正确做法 |
|----------|----------|
| 在 Review 会议上才把PRD拿出来，考大家临场发挥 | 提前一天甚至两天发给所有参与者 |
| 希望靠临场发挥保护"秘密" | 让大家带着问题来开会，线下提前解决 |

> **不要玻璃心**：被程序员吐槽是正常的，不要因此藏着掖着。

---

## 总结

| 章节 | 核心要点 |
|------|----------|
| 版本说明 | 跟着 Review 走，记录变更 |
| 背景与目标 | 解释为什么做 + 设定具体指标 |
| 故事介绍 | 场景描述 → 价值分析 → 用户路径 → 漏斗目标 → 路径规划 |
| 概要设计 | 模块图 + 功能清单（索引）+ 页面结构（自洽） |
| 详细设计 | 与概要设计一一对应，再次自洽 |
| 交付设计 | 数据买点（提前想好）+ 上线筹备（依赖/数据处理） |

**框架建议**：把框架列好，做填空题，至少不会遗漏。设计质量可以慢慢提高，但不遗漏是基本要求。不遗漏 → 有机会复盘和迭代 → 变得更强。
