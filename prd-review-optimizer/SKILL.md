# PRD Review Optimizer

## 目标

以技能内置 `template/PRD指南.md` 为唯一基准，审查用户 PRD 的**文档结构、内容完整性与表达清晰度**；**先给出调整方案供用户确认**，确认后再落地修改。**不改变产品功能定义、业务逻辑或需求范围。**

## 何时使用

- 用户 @ 本技能，或要求审查 / 优化 / 规范化 PRD
- 需对照团队模板检查章节缺失、表述模糊、索引不自洽等问题
- 需补充模板要求的结构性内容（如漏斗目标、交付设计、自洽检验说明）

**不适用**：

- 用户要的是**技术实现方案**（前后端设计、文件清单）→ 使用 `plan-with-story`
- 用户要**新增 / 修改产品功能**（改需求、改交互逻辑）→ 超出本技能范围，应单独讨论
- 用户未提供 PRD 路径或正文 → 先询问，不审查

## 调整边界（必读）

| 允许（结构 & 内容补充） | 禁止（功能调改） |
|-------------------------|------------------|
| 增补缺失章节骨架、调整章节顺序 | 新增 / 删除 / 变更功能点 |
| 补充背景、目标、漏斗、埋点等**占位或引导性**表述 | 修改用户路径、业务规则、验收标准实质 |
| 理顺标题层级、列表、表格、术语统一 | 替用户做产品决策或改 KPI 数值 |
| 标注「待产品确认」的空白项 | 自行脑补未在原文出现的需求 |
| 改写模糊句为更清晰表述（语义不变） | 合并 / 拆分功能模块改变产品 scope |

原文信息不足时：在调整方案中列为「待补充项」，**不擅自填功能内容**。

## 工作流

```
Task Progress:
- [ ] 1. 确认 PRD 来源
- [ ] 2. 读取模板并判断 PRD 类型
- [ ] 3. 对照审查（只读）
- [ ] 4. 输出调整方案 → 等待用户确认
- [ ] 5. 按确认项执行文档优化
```

### 1. 确认 PRD 来源

- 接受：Markdown / 纯文本文件路径，或用户粘贴的正文
- **不猜测路径**；无路径且无正文 → 立即询问
- 无法读取（路径不存在、权限不足）→ 反馈原因，请求重新提供
- `.docx` 等二进制格式 → 请用户导出为 `.md` 或粘贴正文

### 2. 读取模板并判断 PRD 类型

读取 `template/PRD指南.md`，并以文末「总结」表及「自洽检验」段落为检查锚点。

按模板前言判断 PRD 类型，决定检查深度：

| 类型 | 场景 | 检查深度 |
|------|------|----------|
| Index Design | 简单改动（文案、样式等） | 通常无需完整 PRD；若用户仍提交文档，仅做轻量结构建议 |
| Story Design | 单用户故事 / 场景（最常见） | **默认模式**，按六大组成部分审查 |
| Watching Design | 大型产品顶层 PRD | 侧重总纲结构，不要求细节设计与模板同级粒度 |

**选填章节**（路径规划、模块图等）：缺失时列入「建议补充」，**不一律判为错误**。

### 3. 对照审查（只读）

对照模板逐项检查，**此阶段不修改文档、不写完整优化版**：

| 维度 | 检查点（参见模板） |
|------|-------------------|
| 结构 | 版本说明、背景与目标、故事介绍、概要 / 详细 / 交付设计 |
| 故事介绍 | 场景描述、价值分析、用户路径、漏斗目标 |
| 概要自洽 | 功能清单 ↔ 页面结构 ↔ 详细设计 |
| 交付 | 埋点 / 数据分析、上线筹备（依赖、历史数据） |
| 表达 | 标题层级、可量化目标、需求可验证性 |

### 4. 输出调整方案 → 等待用户确认

**必须在此步骤暂停**，等用户明确确认（如「同意」「按方案执行」「排除第 N 项」）后，才能进入步骤 5。

方案仅描述**文档层面**改动，每条标注类型：

- **结构**：增删改章节、调整顺序
- **补充**：补全模板要求但原文缺失的**框架性**内容（留「待确认」占位）
- **表述**：润色、消歧、格式统一（不改变功能语义）

```markdown
## PRD 调整方案（待确认）

**文档**：`path/to/prd.md`
**PRD 类型**：Story Design
**总体符合度**：…（一句话）

**审查摘要**：
- 结构：…
- 内容完整度：…
- 自洽性：…

**拟调整项**：

| # | 类型 | 章节 | 现状问题 | 拟调整（仅结构/补充/表述） | 是否涉及功能 |
|---|------|------|----------|---------------------------|--------------|
| 1 | 结构 | 故事介绍 | 缺少漏斗目标 | 新增「漏斗目标」小节骨架 + 待填写占位 | 否 |
| 2 | 补充 | 交付设计 | 无埋点表 | 按模板补表格列头，行内留空待产品填 | 否 |
| 3 | 表述 | 背景与目标 | 目标不可量化 | 建议将「提升活跃」改为可度量表述（需您确认具体指标） | 否（指标值由您定） |

**明确不做**：
- 不新增 / 删除 / 变更任何功能点
- 不修改原文已确定的用户路径与业务规则

**请确认**：同意全部 / 同意部分（注明序号）/ 需要修改方案
```

用户仅确认「审查结果」但未授权执行 → **仍停在步骤 4**，不写入文件。

#### 何种回复视为「确认执行」

以下表述**视为对已呈现调整方案的确认**，可进入步骤 5（默认执行方案中的**全部**拟调整项，且仍遵守「不做功能调改」边界）：

| 用户表述 | 处理方式 |
|----------|----------|
| 「直接改」「直接优化」「按方案改」「执行吧」 | 视为确认全部拟调整项，开始执行 |
| 「同意」「可以」「OK」「确认」 | 视为确认全部拟调整项（须针对**当前这份**调整方案，而非泛泛认可） |
| 「同意 #1、#3」「只做第 2 项」 | 视为部分确认，仅执行指定序号 |
| 「直接改，但不要动 XX 章节」 | 视为确认，排除指定范围后执行其余项 |

以下**不视为**确认执行，**仍停在步骤 4**：

| 用户表述 | 原因 |
|----------|------|
| 「审查得不错」「问题找得对」 | 仅认可审查，未授权修改 |
| 「我再看看」「先不改」 | 明确暂缓 |
| 提出新的功能需求或改 scope | 超出文档优化边界，需重新审查并出新方案 |

**首次调用且跳过方案**：用户在一开始就要求「直接优化 PRD」且已提供文档 → 仍须先完成步骤 3–4 输出调整方案；用户在同轮或下一轮回复「直接改 / 同意 / 执行」后，方可进入步骤 5。**不得**因用户最初说了「直接优化」而省略方案环节。

**写入文件**：用户说「直接改」默认仅输出优化内容；若需写入磁盘，须额外出现「写入文件 / 保存到 XX 路径」等明确指令。

### 5. 按确认项执行文档优化

用户确认后：

- 只执行方案中**已确认**的条目
- 输出形式按用户要求二选一或兼有：
  - **diff 式说明**：逐条展示修改前后片段
  - **完整优化版 PRD**：基于原文 + 确认项合并输出
- 用户要求写入文件时，才编辑指定路径；否则以 Markdown 回复

## 输出格式（执行后）

```markdown
## PRD 优化结果

**依据方案**：用户已确认项 #1、#2、…

**已执行调整**：
| # | 类型 | 章节 | 调整说明 |
|---|------|------|----------|

**待产品自行填写**（仅留占位，未替您决策）：
- …

**未改动项**（尊重原文功能定义）：
- …
```

## 原则

- **方案先行**：未经用户确认，不输出完整优化版、不修改文件；「直接改 / 同意 / 执行」等视为对**当前方案**的确认，而非跳过方案
- **文档边界**：只做结构、内容补充与表述优化，**不做功能调改**
- **模板唯一**：基准仅为 `template/PRD指南.md`，不引入外部主观标准
- **类型适配**：按 Index / Story / Watching 调整审查深度；选填项不强判
- **保留原意**：符合模板前提下尽量保留原文；不确定处标「待确认」
- **全程中文**：与用户交互使用简体中文；路径、术语、表格内容保持原文习惯
