# Plan with Story

## 目标

在动手写代码前，按 **Spec Coding（规格先行、结构化驱动）** 产出可评审、可落地的方案文档：明确业务目标与边界，识别前后端/数据范围，给出实现步骤与文件改动清单。

## 何时使用

- 用户 @ 本技能，或要求「先出方案 / 设计文档 / 实现计划」
- 新功能、较大改动、跨前后端需求，需要先对齐再编码
- 需要功能示意图、接口规格、文件级改动表

**不适用**：单行 bug 修复、纯问答、用户已给出完整 spec 仅需直接实现。

## 工作流

```
Task Progress:
- [ ] 1. 理解需求与边界
- [ ] 2. 识别设计范围（前端 / 后端 / 数据）
- [ ] 3. 输出结构化设计
- [ ] 4. 拆分实现步骤与检查点
- [ ] 5. 汇总文件清单与约束
```

### 1. 理解需求与边界

- 明确：业务目标、涉及角色、核心流程、边界条件
- 用 1–2 段概括：解决什么问题、如何拆解、技术选型或复用策略
- 给出 **功能示意图** 或 **功能列表**（二选一或兼有）
- 文档结构精炼、逻辑清晰；**全程中文**与用户交互，命令输出可保留原文

### 2. 识别设计范围

根据需求判断涉及哪些层，**只展开相关章节**，跳过无关部分：

| 范围 | 触发信号 |
|------|----------|
| 后端 | API、Service、数据库、权限、定时任务 |
| 前端 | 页面、组件、路由、交互、权限显隐 |
| 数据分析 | 报表、指标、ETL、查询口径 |

涉及代码库时，先快速了解现有模块与命名约定，优先 **复用** 而非新建。

### 3. 结构化设计

#### 后端（如涉及）

**模块定义 / 复用**
- 列出具体文件/类名（如 `UserController.java`）
- 标注 **新增** / **复用**（含路径）；拆分或重组须说明原因与影响

**实现说明**（每模块）
- **接口**：HTTP 方法、路径、请求/响应结构（含示例）
- **核心逻辑**：Service/Manager 调用链与关键方法
- **数据存储**：表、缓存、MQ；对应实体或 Mapper
- **异常**：错误码、异常类与处理方式

**权限**（如涉及）
- 权限码（如 `user:create`）、命名规则、层级（菜单/按钮/接口）
- 与现有权限体系的扩展方式

**其他考虑**
- 迁移脚本（如 `V1.1__add_order_table.sql`）、配置变更
- 缓存策略、定时任务、第三方依赖、性能与安全

#### 前端（如涉及）

**模块定义 / 复用**
- 页面、组件、Hooks/Composables 具体文件名
- 标注新增 / 复用 / 修改及原因

**实现说明**（每页面/组件）
- **路由**：路径、参数、懒加载
- **数据交互**：接口、状态管理、数据流
- **交互**：校验、加载/空/错误态、确认框
- **权限**：按钮/菜单显隐（对应后端权限码）

**UI 草图**
- ASCII 线框标注布局与关键控件；必要时多状态（加载、空、错误）

```
+---------------------------------------------------+
| [← 返回]  订单列表                    [ + 新增 ]  |
+---------------------------------------------------+
| 搜索: [________]  [🔍 搜索]                       |
+---------------------------------------------------+
| 订单号 | 客户 | 金额   | 状态   | 操作            |
| O001   | 张三 | 100.00 | 待支付 | [详情] [删除]   |
+---------------------------------------------------+
| 分页: < 1  2  3 >                                 |
+---------------------------------------------------+
```

**其他考虑**
- i18n 键名、主题变量、移动端适配、UI 库封装、前端缓存、无障碍

### 4. 实现步骤与检查点

按开发顺序拆分；用户未要求测试时 **不增加测试文件**，以手工验证为检查标准。可按实际增减：

| 步骤 | 检查点 |
|------|--------|
| 数据库设计与迁移 | 表创建成功，基础数据可写入 |
| 后端接口开发 | 接口响应正常，校验与异常码正确 |
| 前端页面与联调 | 布局与草图一致，数据与空/错态正常 |
| 权限配置与联调 | 角色下菜单、按钮、接口权限符合设计 |
| 集成验收 | 主流程跑通，边界条件处理正确 |

### 5. 文件清单与约束

**前端文件**

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 新增 | | |
| 修改 | | |
| 删除 | | |

**后端文件**

| 操作 | 文件路径 | 说明 |
|------|----------|------|
| 新增 | | |
| 修改 | | |
| 删除 | | |

**功能约束**
- **边界**：如最多创建 10 条，超出提示
- **非功能**：如列表响应 < 200ms
- **兼容性**：如兼容 App v2.0+ 接口格式
- **一致性**：如删除主记录需同步清理关联数据

## 输出格式

```markdown
## 方案概述
[1–2 段：目标、拆解、选型/复用策略]

## 功能范围
[功能列表或示意图]

## 设计详情
[按 2 识别的范围展开后端/前端/数据章节]

## 实现步骤
[带检查点的步骤表]

## 文件清单
[前后端改动表]

## 约束说明
[边界、非功能、兼容性、一致性]
```

## 原则

- **规格先行**：方案确认后再编码；方案中文件路径、接口、权限须具体到可执行
- **按需展开**：无前端则跳过前端章节，避免模板填空
- **复用优先**：现有模块能覆盖则不新建；改动注明影响范围
- **可验证**：每步有明确检查点，不默认编写用户未要求的测试
- **全程中文**：与用户交互使用简体中文；路径、类名、命令保持原文
